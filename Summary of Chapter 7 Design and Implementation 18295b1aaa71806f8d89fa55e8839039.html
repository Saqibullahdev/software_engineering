Summary of Chapter 7: Design and Implementation

Chapter 7: Design and Implementation - Detailed Summary

This chapter focuses on the design and implementation stages of software engineering, where an executable software system is developed. It covers various topics, including object-oriented design, design patterns, implementation issues, and open-source development. Here's a detailed breakdown:

1. Software Design and Implementation

  • Design is a creative process where software components and their relationships are identified based on customer requirements.
  • Implementation is the process of turning the design into a working program.
  • These two activities are often interleaved, meaning they happen simultaneously rather than one after the other.

Example: If you're building a weather app, the design phase involves deciding what features the app will have (e.g., displaying temperature, humidity, etc.), while implementation involves writing the code to make those features work.

2. Build or Buy

  • In many cases, it’s possible to buy off-the-shelf systems (COTS) that can be customized to meet user requirements, rather than building a system from scratch.
  • This approach can be faster and cheaper.

Example: Instead of building a hospital records system from scratch, you can buy an existing system and customize it to fit the hospital's needs.

3. Object-Oriented Design Using UML

  • Object-oriented design involves creating models of the system using the Unified Modeling Language (UML).
  • For large systems, these models are crucial for communication between different development teams.

Example: If you're designing a weather station, you might create a UML diagram showing how different components (like sensors and data transmitters) interact.

4. Object-Oriented Design Process

  • The design process typically involves:
    1. Defining the system's context and how it will be used.
    1. Designing the system architecture.
    1. Identifying the main objects in the system.
    1. Developing design models.
    1. Specifying object interfaces.

Example: For a weather station, you would first define its purpose (e.g., collecting weather data), then design how the system will be structured, identify objects like sensors and data collectors, and finally specify how these objects will interact.

5. System Context and Interactions

  • Understanding how the system interacts with its environment is crucial.
  • A context model shows the system's relationship with other systems, while an interaction model shows how the system behaves when used.

Example: A weather station interacts with a satellite to send data. The context model shows this relationship, while the interaction model shows how data is transmitted when requested.

6. Architectural Design

  • Once the system's interactions are understood, the next step is to design the system's architecture.
  • This involves identifying major components and organizing them using patterns like layered or client-server models.

Example: A weather station might have separate subsystems for data collection, data transmission, and data storage, all communicating through a common infrastructure.

7. Object Class Identification

  • Identifying object classes is a key part of object-oriented design.
  • There’s no fixed rule for this; it depends on the designer's experience and domain knowledge.
  • Common approaches include:
    • Grammatical approach: Analyzing the system description for nouns (which often represent objects).
    • Behavioral approach: Identifying objects based on their behavior in the system.

Example: In a weather station, objects might include a thermometer, anemometer (wind speed sensor), and barometer (pressure sensor).

8. Design Models

  • Design models show the relationships between objects and classes.
  • There are two main types:
    • Structural models: Show the static structure of the system (e.g., class diagrams).
    • Dynamic models: Show how objects interact over time (e.g., sequence diagrams).

Example: A sequence diagram for a weather station might show how data is collected, summarized, and transmitted to a weather information system.

9. Design Patterns

  • Design patterns are reusable solutions to common problems in software design.
  • They provide a template for solving problems in a way that can be adapted to different situations.

Example: The Observer pattern is used when multiple displays need to show the state of an object. For instance, in a weather station, multiple displays might show temperature, humidity, and wind speed, all updating when new data arrives.

10. Implementation Issues

  • Implementation involves more than just writing code. Key issues include:
    • Reuse: Using existing components or systems to save time and effort.
    • Configuration management: Keeping track of different versions of software components.
    • Host-target development: Developing software on one computer (host) and running it on another (target).

Example: When building a mobile app, you might reuse a library for handling user authentication, manage different versions of the app using a configuration management tool, and test the app on a mobile device (target) while developing it on a PC (host).

11. Reuse

  • Reusing existing software can save time and money.
  • There are different levels of reuse:
    • Abstraction level: Reusing design ideas or patterns.
    • Object level: Reusing objects from a library.
    • Component level: Reusing collections of objects.
    • System level: Reusing entire systems.

Example: Instead of writing a new database system, you might reuse an existing one like MySQL.

12. Configuration Management

  • This involves managing changes to software components during development.
  • Key activities include:
    • Version management: Keeping track of different versions of components.
    • System integration: Combining components to create a working system.
    • Problem tracking: Managing bug reports and fixes.

Example: Using tools like Git for version control to manage different versions of a software project.

13. Host-Target Development

  • Most software is developed on one computer (host) but runs on another (target).
  • The development platform (host) usually has different software than the execution platform (target).

Example: Developing a mobile app on a PC (host) and testing it on a smartphone (target).

14. Open Source Development

  • Open source development involves publishing the source code of a software system and inviting volunteers to contribute.
  • It’s based on the idea that source code should be freely available for users to examine and modify.

Example: The Linux operating system is a well-known open-source product.

15. Open Source Issues

  • Key questions include:
    • Should you use open-source components in your product?
    • Should you develop your software using an open-source approach?

Example: A company might use open-source components to save development time but must ensure they comply with the licensing terms.

16. Open Source Licensing

  • Open-source software is often distributed under specific licenses, such as:
    • GPL (General Public License): Requires that any software using GPL-licensed code must also be open source.
    • LGPL (Lesser General Public License): Allows linking to open-source code without requiring the new software to be open source.
    • BSD License: Allows open-source code to be used in proprietary systems without requiring the new system to be open source.

Example: If you use GPL-licensed code in your project, you must also make your project’s source code available to others.

17. License Management

  • Companies need to manage open-source components carefully, ensuring they comply with licensing terms and track which components are used.

Example: A company might use a tool to track all open-source components in their software and ensure they are used in compliance with their licenses.

Simplified Explanation of Use Cases:

  • A use case is like a story that describes how a system is used. It answers questions like:
    • Who is using the system? (Actors)
    • What do they want to do? (Use case)
    • How does the system respond? (Response)

Example: In a weather station system:

  • Actor: Weather information system
  • Use Case: Request weather data
  • System Response: Send summarized weather data (temperature, wind speed, etc.)

Use Case: Add Item to Cart

  • Actors: Customer, Shopping System
  • Description: The customer adds an item to their shopping cart.
  • Stimulus: The customer selects an item and clicks "Add to Cart."
  • Response: The system updates the cart and displays the updated cart contents.
