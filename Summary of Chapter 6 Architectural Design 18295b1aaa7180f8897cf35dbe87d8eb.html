Summary of Chapter 6: Architectural Design

1. What is Architectural Design?

Architectural design is about organizing a software system and designing its overall structure. It bridges the gap between requirements engineering (what the system should do) and design (how the system will do it). The output is an architectural model that describes how the system is organized into components and how they communicate.

  • Example: Think of designing a house. You need to decide where the rooms (components) will be, how they connect (relationships), and how people (data) will move between them.

2. Agility and Architecture

  • Agile processes often start with designing an overall system architecture.
  • Changing the architecture later can be expensive because it affects many components.
  • Example: If you decide to change the layout of a house after building it, it will be costly and time-consuming.

3. Architectural Abstraction

  • Architecture in the small: Deals with the architecture of individual programs (how a single program is broken into components).
  • Architecture in the large: Deals with complex enterprise systems that may include multiple programs, distributed across different computers, possibly managed by different companies.
  • Example:
    • Small: A single app like a calculator.
    • Large: A banking system with multiple apps, databases, and servers.

4. Advantages of Explicit Architecture

  1. Stakeholder communication: Helps stakeholders (e.g., clients, developers) discuss the system.
  1. System analysis: Allows checking if the system meets non-functional requirements (e.g., performance, security).
  1. Large-scale reuse: The architecture can be reused across multiple systems.
  • Example: A modular architecture for an e-commerce site can be reused for different online stores.

5. Architectural Representations

  • Block diagrams: Simple diagrams showing components and relationships. They are easy to understand but lack detail.
  • Box and line diagrams: Abstract, useful for communication but don’t show detailed relationships or properties.
  • Example: A block diagram of a car might show the engine, wheels, and seats, but not how they are connected in detail.

6. Architectural Design Decisions

Design decisions affect the system’s non-functional characteristics (e.g., performance, security). Common decisions include:

  • How to structure the system.
  • How to distribute the system across hardware.
  • What architectural patterns to use.
  • Example: Deciding whether to use a centralized database (for security) or a distributed one (for scalability).

7. Architecture Reuse

  • Systems in the same domain often have similar architectures.
  • Application product lines: A core architecture with variants for different customer needs.
  • Architectural patterns: Reusable solutions to common problems.
  • Example: Many e-commerce sites use the Model-View-Controller (MVC) pattern to separate data, user interface, and control logic.

8. Architectural Patterns

  • MVC (Model-View-Controller): Separates data (Model), user interface (View), and control logic (Controller).
    • Example: A web app where the database (Model), the webpage (View), and the server logic (Controller) are separate.
  • Layered Architecture: Organizes the system into layers, each providing services to the layer above.
    • Example: An operating system with layers for hardware, kernel, and user interface.
  • Repository Architecture: Central database shared by all components.
    • Example: An Integrated Development Environment (IDE) where tools share a common repository of code.
  • Client-Server Architecture: Servers provide services (e.g., database, printing) to clients.
    • Example: A film library system where clients access a central server for movie information.
  • Pipe and Filter Architecture: Data flows through a series of processing components (filters).
    • Example: A payment system where data flows through stages like invoice processing and payment reminders.

9. Application Architectures

  • Generic application architectures: Common structures for types of systems (e.g., transaction processing, language processing).
  • Use of application architectures: As a starting point for design, a checklist, or a way to organize development.
  • Example:
    • Transaction processing systems: Handle user requests to update a database (e.g., an ATM system).
    • Language processing systems: Process formal language inputs (e.g., a compiler that translates code into machine language).

10. Examples of Application Types

  • Transaction processing applications: Handle user requests and update databases (e.g., online banking).
  • Language processing systems: Process and interpret formal languages (e.g., compilers, interpreters).
  • Example:
    • Transaction processing: An online store where users place orders, and the system updates the inventory.
    • Language processing: A compiler that translates Java code into machine code.

11. Information Systems Architecture

  • Typically organized as a layered architecture with layers like user interface, user communications, information retrieval, and database.
  • Example: A hospital management system with layers for patient data, user authentication, and reporting.

12. Web-Based Information Systems

  • Modern systems are often web-based, with user interfaces in web browsers.
  • Example: E-commerce systems where users add items to a shopping cart and pay online.

13. Language Processing Systems

  • Accept natural or artificial language as input and generate another representation (e.g., machine code).
  • Example: A compiler that translates high-level code (like Python) into machine code that a computer can execute.

14. Compiler Components

  • Lexical analyzer: Converts input code into tokens.
  • Syntax analyzer: Checks the structure of the code.
  • Semantic analyzer: Checks the meaning of the code.
  • Code generator: Produces machine code.
  • Example: A Java compiler breaks down code into tokens, checks syntax, and generates bytecode.

Key Takeaways

  • Architectural design is about organizing a system into components and defining their relationships.
  • Architectural patterns (like MVC, Layered, Client-Server) provide reusable solutions to common problems.
  • Application architectures (e.g., transaction processing, language processing) help design systems for specific needs.
  • Examples: Think of architectural design as building a house, where you decide the layout, rooms, and connections between them. Patterns like MVC are like blueprints that can be reused for different types of houses.
