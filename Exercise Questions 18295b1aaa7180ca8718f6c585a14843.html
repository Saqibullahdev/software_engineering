Question 6.3: Explain why design conflicts might arise when designing an architecture for which both availability and security requirements are the most important non-functional requirements.

Answer:
Design conflicts arise between
availability (the system is always up and running) and security (the system is protected from threats) because:

  1. Security Measures Can Reduce Availability:
    • Adding security layers (e.g., authentication, encryption) can slow down the system or cause downtime during updates.
    • Example: A bankâ€™s website might go offline temporarily for security updates, reducing availability.
  1. High Availability Can Compromise Security:
    • To ensure the system is always available, you might skip security checks or updates, making the system vulnerable.
    • Example: A social media platform might delay security patches to avoid downtime, leaving it open to attacks.
  1. Balancing Act:
    • Achieving both high availability and high security requires careful planning and trade-offs.
    • Example: A cloud service provider might use redundant servers for availability but implement strict access controls for security.

Question 6.5: Why use several architectural patterns in large systems? What additional information is useful?

Answer:

  1. Why Use Multiple Patterns?
    • Complexity of Large Systems: Large systems often have multiple subsystems with different requirements. A single pattern may not address all needs.
      • Example: A banking system might use:
        • Layered Architecture for separating UI, business logic, and database.
        • Client-Server Architecture for handling user requests.
        • Repository Architecture for managing customer data.
    • Flexibility: Different patterns provide flexibility to handle various aspects like scalability, security, and performance.
      • Example: A social media app might use MVC for the frontend and Microservices for backend scalability.
    • Reusability: Patterns allow reuse of proven solutions for common problems.
      • Example: Using Repository Pattern for data access across multiple modules.
  1. Additional Information for Designing Large Systems:
    • Non-Functional Requirements: Performance, security, scalability, and availability.
    • Stakeholder Needs: Understanding what users, managers, and developers need.
    • Technology Stack: Choosing the right tools, frameworks, and platforms.
    • Integration Requirements: How the system will interact with other systems.
    • Regulatory Compliance: Ensuring the system meets legal and industry standards.
    • Cost and Time Constraints: Balancing quality with budget and deadlines.

Question 6.6: Suggest an architecture for a music distribution system like iTunes.

Answer:Architecture for a Music Distribution System:

  1. Layered Architecture:
    • Presentation Layer: User interface for browsing, purchasing, and playing music.
    • Application Layer: Handles business logic like user authentication, payment processing, and music recommendations.
    • Data Layer: Stores music files, user data, and transaction records.
  1. Client-Server Architecture:
    • Client: The iTunes app or web interface used by customers.
    • Server: Backend servers that handle requests, process payments, and stream music.
  1. Repository Architecture:
    • A central database stores all music files, user profiles, and purchase history.
  1. Microservices Architecture:
    • Different services handle specific tasks:
      • User Service: Manages user accounts.
      • Payment Service: Processes transactions.
      • Music Service: Handles music storage and streaming.

Architectural Patterns:

  • Layered Pattern: For separating concerns (UI, logic, data).
  • Client-Server Pattern: For handling user requests and server responses.
  • Repository Pattern: For centralized data storage.
  • Microservices Pattern: For scalability and flexibility.

Question 6.8: Design a system that translates natural language commands into SQL queries.

Answer:Architecture for a Natural Language to SQL System:

  1. Input Module:
    • Accepts natural language commands (e.g., "Show all customers from New York").
  1. Lexical Analyzer:
    • Breaks the input into tokens (e.g., "Show", "customers", "New York").
  1. Syntax Analyzer:
    • Checks the structure of the command (e.g., verb-object-location).
  1. Semantic Analyzer:
    • Maps tokens to database concepts (e.g., "customers" â†’ Customers table, "New York" â†’ city = 'New York').
  1. Query Generator:
    • Converts the analyzed command into an SQL query (e.g., SELECT * FROM Customers WHERE city = 'New York').
  1. Database Interface:
    • Executes the SQL query and returns results.
  1. Output Module:
    • Displays the results to the user.

  • Output SQL: SELECT * FROM Orders WHERE amount > 100.
